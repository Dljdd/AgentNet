01Architecture Overview

How every component connects. No central coordinator — agents discover each other via ENS, coordinate via 0G DA, store data on 0G Storage, run inference on 0G Compute, and serve scores to a Uniswap v4 hook.

System Flow
INDEXER crawls on-chain agent activity → writes structured logs to 0G Storage → broadcasts "new data available" to 0G DA
SCORER subscribes to 0G DA → pulls indexed data from 0G Storage → runs LLM on 0G Compute → writes score + assessment to 0G Storage → broadcasts "new score" to 0G DA
WATCHDOG subscribes to 0G DA → detects anomalies in indexed data → broadcasts signed warnings to 0G DA → scorer adjusts scores based on flags
ROUTER subscribes to 0G DA → maintains a capabilities index from ENS metadata → responds to discovery queries ("find me a reliable oracle agent")
ORACLE subscribes to 0G DA for new scores → pushes finalized trust scores on-chain to Trust Oracle contract → Uniswap v4 Hook reads from Trust Oracle in beforeSwap

ENS is the identity layer across everything — every agent has a *.agentnet.eth subname. Discovery, coordination, hook gating, and the explorer UI all resolve through ENS.

02The Agent Swarm — 5 Types, No Coordinator

Each agent operates independently. They discover each other by resolving ENS subnames, communicate over 0G DA, and share data through 0G Storage. No agent has authority over another — consensus emerges from independent action.

🔍 INDEXER AGENT
Purpose: On-chain data ingestion.
Crawls agent registrations, monitors Uniswap pool interactions, tracks transaction history. Publishes structured activity logs to 0G Storage. Announces new data availability on 0G DA so other agents can react.

Inputs: On-chain events (agent registry, Uniswap pools)
Outputs: Structured activity records on 0G Storage
Triggers: New block / new agent registration / new swap
indexer.agentnet.eth
🧠 SCORER AGENT (LLM-POWERED)
Purpose: Trust assessment via AI inference.
Consumes indexed activity from 0G Storage. Runs an LLM on 0G Compute to analyze transaction patterns and produce both a numeric score (0–100) and a natural-language assessment. Multiple scorer instances run independently — final score is the median consensus.

Inputs: Activity records from 0G Storage, watchdog flags
Outputs: Trust score + written reasoning on 0G Storage
Triggers: "new data available" messages on 0G DA
scorer-01.agentnet.eth
🚨 WATCHDOG AGENT
Purpose: Anomaly detection and security.
Monitors indexed data for suspicious patterns — wash trading (same agent trading with itself through proxies), sybil clusters (many agents with correlated behavior), sudden volume spikes, and rug-pull indicators. Broadcasts signed warning messages via 0G DA that scorer agents factor into their assessments.

Inputs: Activity records from 0G Storage
Outputs: Signed alerts on 0G DA, flag records on 0G Storage
Triggers: Periodic scans + "new data available" events
watchdog.agentnet.eth
🗺️ ROUTER AGENT
Purpose: Agent discovery and matchmaking.
Maintains a live capabilities index by reading ENS text records for all registered agents. Responds to discovery queries from other agents or the MCP server — e.g. "find me a high-trust oracle agent" or "which agents are active on this Uniswap pool?" Enables composability without hardcoded agent addresses.

Inputs: ENS text records, trust scores from 0G Storage
Outputs: Ranked agent recommendations
Triggers: Inbound queries via 0G DA or MCP
router.agentnet.eth
🔗 ORACLE AGENT
Purpose: Bridge between off-chain swarm and on-chain contracts.
Reads finalized trust scores from 0G Storage and pushes them to the on-chain Trust Oracle contract that the Uniswap v4 hook queries. Monitors for stale scores (>N blocks old) and triggers re-evaluation by broadcasting to scorer agents via 0G DA. Also handles batch updates to minimize gas costs.

Inputs: Trust scores from 0G Storage, "new score" events on 0G DA
Outputs: On-chain Trust Oracle updates (Ethereum Sepolia)
Triggers: "new score" events + staleness timer
oracle.agentnet.eth
// Agent Communication Pattern
All inter-agent communication flows through 0G DA as a decentralized pub/sub bus:
indexer.agentnet.eth publishes → topic: agent-activity
scorer-*.agentnet.eth subscribes → topic: agent-activity & topic: watchdog-alerts
watchdog.agentnet.eth publishes → topic: watchdog-alerts
oracle.agentnet.eth subscribes → topic: trust-scores
router.agentnet.eth subscribes → topic: * (maintains full state)
No agent calls another agent directly. All coordination happens via topics on 0G DA. If any agent goes down, the others continue operating — true decentralization.

03The AI Angle — LLM-Powered Scoring

This is what separates AgentNet from "a distributed system with agent branding." The Scorer doesn't compute a formula — it reasons about agent behavior.

🧠 HOW THE SCORER THINKS
The Scorer agent pulls an agent's full activity history from 0G Storage, formats it into a structured prompt, and sends it to an LLM running on 0G Compute. The LLM produces:
NUMERIC SCORE
0–100 trust score based on: transaction success rate, counterparty diversity, volume consistency, pool breadth, age, watchdog flag count. This is what the Uniswap hook reads.
WRITTEN ASSESSMENT
Natural-language reasoning explaining why this agent got this score. Stored on 0G Storage, displayed in the explorer. This is auditable, transparent AI — not a black box.
EXAMPLE SCORER OUTPUT
Trust Score: 84/100 — HIGH TRUST
"This agent has executed 47 successful swaps across 12 Uniswap pools over 18 days with consistently low slippage (<0.3%). Transaction patterns show diversified counterparties with no wash-trading indicators. Volume is steady without suspicious spikes. No watchdog flags on record. The agent's interaction graph shows connections to 8 other high-trust agents, suggesting established network participation. Recommended for standard and elevated DeFi interactions."
— scorer-01.agentnet.eth · via 0G Compute · 2 min ago
This is the demo's "wow moment." When a judge clicks a trust score in the explorer and sees this reasoning appear — that's when they understand this is genuinely AI-powered.
WATCHDOG INTELLIGENCE
The Watchdog agent also uses pattern analysis (can be LLM-assisted or heuristic) to detect:

WASH TRADING Agent swapping with itself through proxy addresses — detected by analyzing counterparty graph clustering
SYBIL PATTERNS Multiple agents with suspiciously correlated activity (same timing, same pools, same volumes)
RUG INDICATORS Sudden large withdrawals after building trust, or abandonment of LP positions
VOLUME SPIKES Abnormal transaction volume that could indicate gaming the trust score

04ENS Identity Layer

Every agent gets a human-readable name. ENS isn't bolted on — it's the identity infrastructure that everything else resolves through.

🪪 SUBNAME SCHEMA: *.agentnet.eth
Each agent registered in the network receives a subname under agentnet.eth. System agents have fixed names (indexer.agentnet.eth, oracle.agentnet.eth). External agents registering for trust scoring receive names like agent-0x4a3f.agentnet.eth or can claim custom names.
4 Integration Points
1. AGENT DISCOVERY
Agents find each other by resolving ENS subnames. oracle.agentnet.eth always points to the current oracle contract, regardless of upgrades or redeployment. No hardcoded addresses anywhere.
2. METADATA VIA TEXT RECORDS
ENS text records store: role, trust-tier, 0g-storage-key, capabilities, operator. The explorer and router agent read directly from ENS — no separate database.
3. ACCESS GATING IN HOOK
The Uniswap v4 hook checks ENS reverse resolution to confirm the caller has a valid *.agentnet.eth identity before looking up its trust score. No ENS name = no trust-gated access.
4. HUMAN-READABLE EXPLORER
Users search scorer-01.agentnet.eth, not 0x4a3f.... Trust history is anchored to the ENS identity. Even if an agent's underlying contract changes, the name and reputation persist.

05Uniswap v4 Trust-Gated Hook

A beforeSwap hook that reads trust scores from the on-chain oracle and adjusts fees by tier. Simple, functional, and exactly what Uniswap asked for: "reliability, transparency, composability."

Agent calls swap() → ENS reverse lookup: *.agentnet.eth? → beforeSwap reads Trust Oracle → 80+ → reduced fee (0.1%) | 40–79 → standard fee (0.3%) | <40 → elevated fee (1%) or revert
The hook itself is intentionally simple — all intelligence lives in the agent swarm. The hook is a pure consumer: it reads one value (trust score) and applies a fee multiplier. This keeps the on-chain surface area minimal and auditable.
HOOK CONTRACT
Implements beforeSwap on Uniswap v4. Reads msg.sender → ENS reverse resolve → query Trust Oracle → return dynamic fee override. Deployed on Sepolia with a real Uniswap v4 pool.
TRUST ORACLE CONTRACT
On-chain mapping of address → uint8 trustScore. Updated by the Oracle agent. Read by the hook. Includes a lastUpdated timestamp so the hook can reject stale scores (configurable threshold).

060G Integration — Full Stack

0G is the backbone. Every agent depends on it for storage, compute, and coordination. This is a genuine full-stack 0G integration — not a surface-level mention.

0G STORAGE
What's stored:
• Agent activity records (from Indexer)
• Trust scores + LLM assessments (from Scorer)
• Watchdog flag records
• Historical reputation snapshots
• Agent registration metadata

Why 0G: Decentralized, no single point of failure. The explorer reads directly from 0G — no backend database.
0G COMPUTE
What runs:
• LLM inference for trust scoring
• Pattern analysis for the Watchdog
• Assessment generation

Why 0G: Verifiably decentralized AI inference. When the explorer shows "scored by scorer-01 via 0G Compute," it means the AI wasn't running on someone's laptop — it ran on decentralized infra.
0G DA
What flows through it:
• "New data available" (Indexer → Scorer)
• "Watchdog alert" (Watchdog → Scorer)
• "New score published" (Scorer → Oracle)
• Discovery queries (Router ↔ anyone)

Why 0G: The coordination backbone. Agents discover data and each other through 0G DA topics — no central message broker.

07Explorer + MCP Server

The human-facing window into the protocol, plus a programmatic API so other agents and developers can query trust data.

🖥️ EXPLORER UI
Search: By ENS name (scorer-01.agentnet.eth) or raw address. ENS-first experience.
Agent Profile: Registration card, trust score (big number), LLM-written assessment (expandable), trust tier badge, operator info, ENS text records.
History: Reputation-over-time chart (line graph), transaction timeline, score change log with reasoning for each change.
Network Graph: D3.js force-directed graph showing counterparty relationships. Node size = trust score. Edge thickness = interaction frequency.
Live Feed: Real-time activity from the swarm — indexer crawling, scorer publishing, watchdog flagging, oracle pushing. Shows the protocol is alive, not static.
Comparison: Side-by-side agent comparison (for the demo: high-trust vs low-trust agent fees).
⚡ MCP SERVER
Programmatic access for other agents and developers. Makes AgentNet composable — any agent can check another agent's reputation before interacting.

getTrustScore(ensName) → returns numeric score + tier
getAssessment(ensName) → returns full LLM-written reasoning
getHistory(ensName, timeRange) → returns reputation timeline
getNetworkGraph(ensName, depth) → returns counterparty graph
discoverAgents(query) → routes to Router agent for capability-based search
getWatchdogAlerts(ensName) → returns any active flags

13Risk Mitigations

Things that could go wrong and how we handle them.

SCOPE CREEP
Day 7 is a hard gate. If the core loop (index → score → oracle → hook) doesn't work, all hands fix it — no new features. Watchdog and Router are built in Days 4–5, before the gate. If they're not ready, they get cut, not delayed.
COLD START
15–20 seeded agents with real transaction histories by Day 7. Mix of trust profiles. Demo shows a new agent entering an alive ecosystem. Judges never see an empty system.
"WHERE'S THE AI?"
Scorer uses a real LLM on 0G Compute. Trust assessments include written reasoning, not just numbers. The demo's wow moment is the LLM assessment streaming in real-time. Watchdog uses pattern analysis. This is genuine AI, not a formula with agent branding.
DASHBOARD FATIGUE
Demo opens with terminals and agents, not the UI. Explorer appears at 2:40 as the payoff — after judges have watched the swarm work for 2+ minutes. Lead with intelligence, end with interface.
HOOK COMPLEXITY
Hook is intentionally simple: beforeSwap → ENS check → read oracle → fee multiplier. All intelligence is in the agents, not the hook. Small on-chain surface area = less that can break.
INTEGRATION GAPS
P4 (Explorer) and P5 (MCP) start with mock data on Day 3 so they're not blocked by agent development. Real data is swapped in when agents are ready. Parallel workstreams, not serial.
DEMO DAY FAILURES
Full dress rehearsal on Day 10. Video recorded on Day 11 (not live). Pre-execute the Uniswap swaps before recording to avoid block confirmation delays. Keep the swarm running for judges who want to test live, but the video is the primary submission.
